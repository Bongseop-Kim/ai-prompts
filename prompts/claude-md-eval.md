# CLAUDE.md 평가 프롬프트

> Claude Code 루트 디렉토리에서 실행. 평가만 수행하며 파일을 수정하지 않는다.

---

## 지시사항

프로젝트 내 모든 CLAUDE.md 파일을 찾아서 아래 기준으로 평가하라.

**실행 순서:**
1. `find . -name "CLAUDE.md" -not -path "*/node_modules/*"` 로 모든 CLAUDE.md 위치 확인
2. 각 파일을 읽어라
3. 파일별로 아래 A~H 기준을 적용해 평가하라
4. 파일을 수정하지 말고 평가 결과와 개선 제안만 출력하라
5. `package.json`, `README.md` 등 관련 파일을 참조해서 stale 여부를 검증하라

---

## 평가 기준

### 기준 A: 중복성 (Redundancy)
에이전트가 소스 코드, README, package.json 등을 읽어서 스스로 파악할 수 있는 내용이 포함되어 있는가?

- 기술 스택 나열 ("This is a Next.js project") → **감점**
- 디렉토리 구조 설명 → **감점**
- 에이전트가 추론 불가능한 고유 정보 (특수 환경 설정, 팀 컨벤션 예외사항) → **가점**

> 근거: 잘 문서화된 레포지토리에서 중복 정보는 비용만 올리고 성능은 낮춘다 (ETH Zürich, 2026).

---

### 기준 B: 액션 가능성 (Actionability)
내용이 실행 가능한 명령어 중심인가, 아키텍처 설명 중심인가?

- `pnpm test`, `npm run lint:fix` 같은 copy-paste 가능한 명령어 → **가점**
- 비자명한 gotcha/quirk 문서화 ("이 프로젝트는 npm 대신 pnpm 사용") → **가점**
- "이 프로젝트는 모노레포 구조입니다" 같은 서술형 설명 → **감점**
- 아키텍처 개요(Architecture Overview) 섹션 → **감점**

> 근거: 에이전트는 지시를 따르지만 아키텍처 설명은 코드 탐색 효율을 높이지 못한다 (ETH Zürich, 2026).

---

### 기준 C: 간결성 및 가독성 (Conciseness & Readability)
문서가 짧고 명료한가?

- 전체 토큰 수 추정 (root: 1,500 이하 권장, package별: 500 이하 권장)
- 불필요한 섹션, 중복 설명 존재 여부
- 지시문이 명확한 명령형인가, 장황한 설명형인가
- 법률 문서 수준의 복잡한 문장 구조 → **감점**

> 근거: 평균 Claude Code context file의 Flesch Reading Ease 점수는 16.6 (법률 문서 수준). 짧고 명료한 파일이 더 안정적으로 준수된다 (Agent READMEs, 2025.11).

---

### 기준 D: Claude 준수 가능성 (Compliance Likelihood)
Claude가 실제로 이 지시를 따를 가능성이 있는가?

- Claude Code 시스템 프롬프트에는 이미 ~50개의 지시사항이 있다. CLAUDE.md는 태스크와 관련 없다고 판단되면 무시된다.
- 현재 태스크와 무관할 가능성이 높은 지시사항 → **감점**
- 모든 세션에 보편적으로 적용 가능한 지시사항 → **가점**
- 특정 작업(DB 스키마 설계 등)에만 해당하는 지시사항은 CLAUDE.md가 아닌 skill이나 slash command로 분리 권장

> 근거: Claude는 CLAUDE.md를 system-reminder로 주입하되, 관련 없다고 판단하면 무시한다 (HumanLayer, 2025).

---

### 기준 E: 검증 가능성 (Verifiability)
문서화된 명령어와 경로가 현재도 유효한가?

- 실제로 존재하는 파일 경로인가 확인
- 명시된 명령어(`package.json`의 scripts와 비교)가 현재도 유효한가 확인
- 기술 스택 정보가 실제 `package.json`, 설정 파일과 일치하는가 확인
- stale 가능성이 높은 정보 (버전 명시, 경로 명시) → **주의 표시**

> 근거: 오래된 빌드 명령이 문서에 있으면 에이전트가 그걸 따르다 실패한다. 유지보수가 진짜 어려운 부분이다 (Packmind, 2026).

---

### 기준 F: 모노레포 계층 일관성 (Cross-file Consistency)
*(해당하는 경우에만 적용)*

여러 CLAUDE.md 간 충돌하거나 중복되는 지시사항이 있는가?

- root CLAUDE.md와 package별 CLAUDE.md 간 모순 지시 → **감점**
- 동일 내용이 여러 파일에 중복 → **감점**
- root에서 전역 규칙, package에서 로컬 예외만 다루는 구조 → **가점**

---

### 기준 G: LLM 자동 생성 흔적 (Auto-generation Indicators)
`/init`이나 LLM으로 자동 생성된 흔적이 있는가?

다음 패턴은 자동 생성 가능성이 높다:
- "This project uses [기술 스택]" 형식의 도입부
- 디렉토리 트리 (`src/`, `components/`, `api/` 등 나열)
- 일반적인 best practice 나열 ("Write clean code", "Follow SOLID principles")
- 과도하게 완성된 구조 (빠진 섹션이 없이 모든 항목이 채워진 형태)

> 근거: LLM 자동 생성 context file은 성능을 낮추면서 비용만 올린다. 단, 기존 문서화가 부족한 레포지토리에서는 예외적으로 유효할 수 있다 (ETH Zürich, 2026).

---

### 기준 H: 경험 기반 여부 (Experience-driven)
실제 에이전트 사용 중 발견된 실수나 마찰에서 비롯된 내용인가, 아니면 예방적으로 작성된 내용인가?

- "에이전트가 반복적으로 npm을 사용해서 추가함" 같은 실제 경험 기반 규칙 → **가점**
- 처음부터 완성형으로 작성된 느낌 → **감점**

> 근거: 빈 파일에서 시작해 에이전트가 반복적으로 틀리는 패턴만 추가하는 방식이 가장 효과적이다 (Upsun, 2026).

## 출력 형식

각 CLAUDE.md 파일에 대해 아래 형식으로 출력하라.

```
### [파일 경로]

| 기준 | 점수 | 근거 |
|------|------|------|
| A. 중복성 | ✅/⚠️/❌ | [구체적 내용] |
| B. 액션 가능성 | ✅/⚠️/❌ | [구체적 내용] |
| C. 간결성/가독성 | ✅/⚠️/❌ | [토큰 추정치 포함] |
| D. Claude 준수 가능성 | ✅/⚠️/❌ | [구체적 내용] |
| E. 검증 가능성 | ✅/⚠️/❌ | [stale 의심 항목 나열] |
| F. 계층 일관성 | ✅/⚠️/❌/N/A | [충돌 항목 나열] |
| G. LLM 생성 흔적 | ✅/⚠️/❌ | [구체적 패턴] |
| H. 경험 기반 여부 | ✅/⚠️/❌ | [판단 근거] |

**종합 등급:** S / A / B / C / D

**즉시 제거 권장 항목:**
- [삭제해야 할 내용과 이유]

**수정 권장 항목:**
- [수정이 필요한 내용과 개선 방향]

**유지 권장 항목:**
- [잘 작성된 부분]
```

---

## 종합 등급 기준

| 등급 | 기준 |
|------|------|
| S | 중복 없음, 명령어 중심, 경험 기반, 검증됨, 간결함 |
| A | 대부분 양호. 1~2개 기준 미흡 |
| B | 기본적으로 유용하나 중복 또는 검증 부재 |
| C | 중복 많음, LLM 자동생성 흔적, "많을수록 좋다" 방식 |
| D | 에이전트 성능에 실질적으로 해로울 수 있는 수준 |

---

## 주의사항

- 파일을 수정하지 말 것
- 평가 결과만 출력할 것
- 각 판단은 파일의 실제 내용을 인용해서 구체적으로 근거를 제시할 것
- `package.json`, `README.md` 등 관련 파일을 참조해서 stale 여부를 검증할 것
