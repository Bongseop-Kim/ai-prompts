## 배점 근거 (논문 기반)

**[근거 1] Sabra et al. (arXiv:2508.14727, 2025)**
LLM 생성 코드는 모델 종류와 무관하게 90~93%가 코드 스멜, 5~8%가 버그,
약 2%가 보안 취약점을 포함한다. 기능 테스트를 통과해도 정적 분석 없이는
프로덕션 부적합.
→ 보안 정적 분석 도구 항목 신설 및 보안 배점 상향의 근거.

**[근거 2] Li et al. IRIS (arXiv:2405.17238, ICLR 2025)**
LLM은 단독으로 취약점 탐지에 실패하지만, 정적 분석(CodeQL)과 결합 시
탐지율이 27건 → 69건으로 2.5배 상승.
→ 린터와 AI 툴의 통합 수준을 별도 영역으로 분리하는 근거.

**[근거 3] GitClear 2025 / arXiv:2510.10165**
AI 도구 도입 후 중복 코드 블록 8배 증가, PR 재작업량 증가. AI 지원 코드는
기술 부채를 축적시키는 체계적 패턴 존재.
→ 중복 코드·기술 부채 지표를 운영 상태 점수에 포함하는 근거.
  단, 이 논문은 현상을 관찰한 것이므로, 도구 도입이 직접적 해결책임을
  의미하지는 않는다. 탐지 가시성 확보 수단으로 평가한다.

## 점검 영역 및 배점 기준 (총 100점)

### 영역 1. 도구 설치 및 기본 설정 (15점)

**린터 설치 및 규칙 파일 (8점)**
- 0점: 린터 없음
- 5점: 린터 설치됨, 규칙 파일 존재
- 8점: 린터 설치됨, 규칙 파일 존재 + extends 또는 plugins로 기본값 이상 확장됨

**포매터 설치 및 설정 파일 (7점)**
- 0점: 포매터 없음
- 4점: 포매터 설치됨, 설정 파일 존재
- 7점: 포매터 설치됨, 설정이 팀 컨벤션(print width, tab 등)을 명시적으로 정의함

### 영역 2. 자동화 수준 (20점)

**pre-commit hook (13점)**
- 0점: hook 없음
- 6점: hook 존재하지만 lint 또는 format 중 하나만 포함
- 10점: lint + format 모두 자동 실행
- 13점: lint + format + type-check 또는 test까지 포함

**CI/CD 파이프라인 lint 강제 (7점)**
- 0점: CI 없거나 lint 미포함
- 4점: CI에 lint 포함되지만 실패 시 통과 가능 (warn 수준)
- 7점: CI에서 lint 실패 시 머지 차단

### 영역 3. AI 코딩 도구 통합 (20점)

> 근거 2 적용: LLM + 정적 분석 결합이 탐지율 2.5배 상승

**AI 도구(Claude Code, Cursor 등) 사용 프로젝트**
- 0점: AI 사용하지만 lint가 피드백 루프에 없음
- 8점: 개발자가 수동으로 lint 실행 (CLAUDE.md 등에 lint 명령어 가이드 있음)
- 14점: pre-commit hook으로 AI 수정 파일에 lint 자동 실행
- 20점: hook + CI 이중 차단 + AI 컨텍스트 파일(CLAUDE.md 등)에 lint 규칙이 명시적으로 참조됨

**AI 도구 미사용 프로젝트**
- 이 항목은 N/A 처리. 나머지 80점을 100점으로 환산하여 최종 점수 산정.
- 환산 공식: `최종점수 = (나머지 5개 영역 합산 / 80) × 100`

### 영역 4. 보안 정적 분석 (20점)

> 근거 1 적용: AI 생성 코드의 ~2%가 보안 취약점, 그 중 60~70%가 BLOCKER급

**보안 전용 정적 분석 도구 존재 여부 (10점)**
- 0점: 없음
- 4점: eslint-plugin-security 수준 (규칙 기반, 경고 위주)
- 7점: Semgrep OSS 등 패턴 매칭 SAST 도구
- 10점: CodeQL 또는 상용 SAST 도구 (Snyk, SonarQube 포함)

**보안 규칙의 적용 범위 (10점)**
- 0점: 로컬에서만 실행되거나 AI 생성 코드에 적용 안 됨
- 5점: PR 단계에서 수동 실행 또는 부분 적용
- 10점: CI에서 모든 커밋/PR에 자동 SAST 실행, 실패 시 머지 차단

### 영역 5. 규칙의 전략적 설계 (15점)

> 린트 규칙이 단순 스타일 가이드가 아니라, AI가 실수하기 쉬운 패턴을
> deterministic하게 차단하는 제약으로 설계되어 있는지 평가

- 0점: 기본값(recommended) 그대로 사용
- 5점: 일부 커스텀 규칙 존재 (프로젝트 특화 off/warn 조정 수준)
- 10점: import path 강제, deprecated API 차단, naming 규칙 등 컨벤션이 인코딩됨
- 15점: 위 조건 + CLAUDE.md 또는 컨텍스트 파일보다 강한 제약으로 설계됨
         (규칙이 soft guideline이 아닌 hard error로 설정됨)

### 영역 6. 운영 상태 및 기술 부채 (10점)

> 근거 3 적용: AI 도구 도입 후 중복 코드 8배 증가 관찰
> 도구 존재 자체보다 "문제가 보이는 상태"인지를 평가

**eslint-disable 남용 (5점)**
- 0점: 10개 이상 (AI 생성 코드 무검증 병합 패턴 의심)
- 2점: 5~9개
- 4점: 1~4개
- 5점: 0개

**중복 코드 탐지 도구 (5점)**
- 0점: 없음
- 2점: jscpd / SonarQube 등 설정 존재하지만 CI 미연동
- 5점: CI에서 자동 실행, 임계값 초과 시 경고 또는 차단
